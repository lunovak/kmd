Please number the tasks in pattern KMD-epicNumber-taskNumber, where the epics are those stories you proposed. Let's start with this epic 2:

Performance Management (Admin)

As an Admin:

I want to create a performance so that members can reserve tickets
I want to edit a performance so that I can correct mistakes
I want to delete a performance so that cancelled events are removed
I want to set ticket capacity so that overbooking is prevented
I want to assign a play and theatre to a performance so that events are well described
I want to define reservation waves so that booking rules are enforced

You picked the part that quietly hides half the system complexity. Bold move. This “simple CRUD” is exactly where projects start pretending to be easy and then slowly mutate.

Let’s turn it into something Copilot can actually chew on without hallucinating half your domain.

🧠 Reality Check

This “one epic” already:

introduces a second entity (ReservationWave)
defines core validation rules that will affect reservations
creates future coupling with notification logic

In other words, this is not CRUD. This is your domain backbone wearing a CRUD costume.

Next step, if you want to avoid rewriting half of this later, is breaking down Reservation Management. That’s where your rules will either hold… or collapse in a very educational way.
