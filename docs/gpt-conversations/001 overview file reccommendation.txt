# Theatre Ticket Reservation App – Overview

## Summary
A private web application for managing theatre ticket reservations within a fixed group of members. An administrator defines performances and available tickets. Members reserve tickets under simple rules. The app is not public-facing.

## Goals
- Manage a fixed set of users (no public registration)
- Admin can create performances and assign ticket limits
- Members can reserve and cancel tickets
- Prevent overbooking
- Send basic email notifications
- Simple, maintainable solution

## Non-Goals
- Payments or billing
- Public sign-up
- Seat-level selection (only ticket counts)
- Mobile app or complex frontend
- Multi-tenant architecture

## Core Concepts
- User (Admin or Member, via ASP.NET Identity)
- Performance (event with date/time and ticket capacity)
- Reservation (user reserves N tickets for a performance)

## Functional Requirements

### Authentication
- Use ASP.NET Core Identity
- Users are pre-created by admin
- Roles: Admin, Member

### Admin
- Create/edit/delete performances
- Set ticket capacity
- View all reservations

### Member
- View performances
- Create/cancel reservations
- View own reservations

### Notifications
- Email on reservation confirmation and important changes

## Business Rules
- Reservations are first-come-first-served
- Total reserved tickets cannot exceed capacity
- Optional: limit tickets per user per performance
- All validation must be enforced server-side

## Technical Stack
- ASP.NET Core Razor Pages
- C#
- Entity Framework Core
- Azure SQL (Free tier)
- ASP.NET Core Identity
- Azure App Service
- Email via SMTP or Azure Communication Services

## Architecture
- Razor Pages (no MVC unless needed)
- Layers:
  - Pages (UI)
  - Services (business logic)
  - Data (EF Core)
- Use dependency injection
- Keep design simple, avoid over-engineering

## Data Model (High-Level)
- User (Identity)
- Performance (Id, Name, DateTime, Capacity)
- Reservation (Id, UserId, PerformanceId, TicketCount)

Relationships:
- Performance → many Reservations
- User → many Reservations

## Security
- Identity for authentication
- Role-based authorization (Admin vs Member)
- Validate all inputs server-side
- Prevent overbooking at database/service level

## Deployment
- Azure App Service
- Azure SQL database
- Configuration via environment variables
- CI/CD via GitHub Actions (preferred)

## Development Guidance
- Prefer small, incremental changes
- Keep project always buildable
- Use EF Core migrations for schema changes
- Generate clear, maintainable code (no unnecessary abstractions)
- Avoid introducing new frameworks without clear need

## Out of Scope
- Payments
- Public APIs
- Real-time updates (e.g., SignalR)
- Complex UI frameworks
